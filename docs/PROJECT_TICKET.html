1 Project Overview
What is MarketPulse?
MVP
MarketPulse is a real-time financial market dashboard that aggregates data across stocks, cryptocurrency, commodities, forex, global indices, and financial news into a single, customisable web interface. Users get a live pulse on global markets through a draggable card-based layout without switching between multiple sites or tools.
🎯 Primary Goal
Provide a free, self-hostable market intelligence hub that replaces paid Bloomberg / TradingView subscriptions for retail investors and developers.
👤 Target Users
Retail investors, day traders, finance students, and developers who want real-time market data without API subscription costs.
💡 Key Constraint
All data APIs must be free-tier. No paid subscriptions. Rate limits are managed via backend caching and request throttling.
2 Technology Stack
🖥️ Backend
.NET 8 ASP.NET Core Entity Framework Core 8 SQL Server (LocalDB dev) JWT Auth (15-min tokens) Swagger / OpenAPI Clean Architecture HttpClient (external APIs)
🌐 Frontend
React 18 TypeScript 5 Vite 5 TailwindCSS 3.4 React Router v7 React Grid Layout Sonner (toasts) localStorage (state)
Architecture Pattern: Clean Architecture (Domain → Application → Infrastructure → API). Frontend communicates through Vite proxy in development (/api → http://localhost:5167). Production will use a reverse proxy (Nginx / IIS).

MP-008 AI Market Insights Card P3 · Future
Source Claude API (free via Anthropic Console credits) OR static rule-based summaries (no API)
Content Daily market summary, top movers narrative, sentiment indicators, notable events
Approach Generate once per day — use Claude API to summarize top headlines + market moves into 5-6 bullet insights. Cache result 24h.
Backend GET /api/insights — returns cached AI-generated summary. Regenerated via scheduled job at market open.
MP-009 Real-Time Price Updates (WebSocket / SSE) P2 · Pending
Sources Binance WebSocket (crypto — free) Finnhub WebSocket (stocks — free)
Approach Backend subscribes to Finnhub & Binance WebSocket streams. Pushes updates to connected clients via Server-Sent Events (SSE) or SignalR. Avoids direct client WebSocket to external APIs (CORS/key security).
Backend GET /api/stream/prices — SSE endpoint. Client subscribes once and receives price ticks for subscribed symbols.
Frontend Use EventSource API to connect to SSE endpoint. Update PriceGrid / IndexGrid components without full re-renders.
MP-010 Backend Response Caching (Rate-Limit Management) P1 · Required
Why All free APIs have low rate limits. Multiple dashboard users hitting the same endpoints would exhaust quotas quickly without caching.
Solution IMemoryCache (.NET built-in — free) OR Redis (Docker, free self-hosted)
TTL Strategy Crypto: 30s | Stocks/Indexes: 60s | Commodities: 5min | News: 15min | Sectors: 5min | Weather: 10min/city | AI Insights: 24h
Implementation Add ICacheService abstraction wrapping IMemoryCache. Inject into all market data services.
MP-011 Global Search Bar (Dashboard Header) P3 · Future
APIs Finnhub Symbol Search (free) CoinGecko coin search (free)
Behaviour Type ticker/name → see matching stocks + crypto. Click result to open a detail modal with live quote + mini-chart.
Backend GET /api/search?q={query} — returns array of {symbol, name, type, exchange}. Debounce 300ms on frontend.

8 Non-Functional Requirements

⚡ Performance

  • Dashboard initial load < 2s on LAN
  • API response < 200ms (from cache)
  • Real-time updates < 1s latency
  • No render blocking on card load failure

🔒 Security

  • All API keys stored server-side only
  • JWT tokens short-lived (15 min)
  • CORS locked to known origins
  • No sensitive data in frontend bundle

♻️ Resilience

  • Each card fails independently (no cascading)
  • Fallback to cached/dummy data on API error
  • Retry with exponential backoff (Polly)
  • Rate limit errors handled gracefully (429 → use cache)

📐 Scalability

  • Backend caching reduces external API calls by 90%+
  • Stateless API (JWT) — horizontally scalable
  • SSE preferred over polling where possible
  • Pagination on news & stock list endpoints

🧩 Maintainability

  • Each API client in its own infrastructure class
  • Feature flags for experimental cards
  • API keys configurable per environment
  • Swagger docs for all endpoints

📱 Usability

  • Dark theme — accessible contrast ratios
  • Draggable layout saved per user
  • Responsive grid (tablet / desktop)
  • Card refresh without full page reload

10 Constraints & Risks
⚠ Free API Rate Limits are the #1 risk. All external data sources operate on free tiers with strict daily/monthly quotas. Server-side caching (Phase 1) is a hard prerequisite before wiring real APIs to the dashboard — without it, a single page refresh can exhaust a daily quota.

⚡ Risk Register
